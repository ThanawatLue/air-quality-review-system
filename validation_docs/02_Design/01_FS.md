SECTION 1: SYSTEM ARCHITECTURE & EXECUTABLE DELIVERY
This Functional Specification (FS) details the technical methods used to fulfill the User Requirements (URS). While this document contains technical programming details, plain English explanations are provided so that all project stakeholders can understand how the system works.

1.1 PyInstaller Implementation
The application is bundled into a standalone binary program (`AQR_Dashboard_v1.1.0.exe`).
•	Path Resolution: The system uses specific code (`sys._MEIPASS`) to find its own hidden internal files (like images or templates) when running, and another command (`os.path.dirname(sys.executable)`) to know exactly where to save your final Excel reports on your computer.
•	Server: It automatically starts a lightweight, hidden web server (`waitress` WSGI server) in the background so you can access the user interface through a standard web browser at port 5000.

SECTION 2: TECHNICAL IMPLEMENTATION DETAILS

2.1 User Interface & Workflow
•	Dashboards: The system provides an 'Analysis' screen for generating reports and a 'Data Transformation' screen for splitting large files.
•	Directory Selection: Users select folders using the screen. Behind the scenes, the system verifies that the folder actually exists using a system check (`os.path.isdir()`).
•	Limit Reading: The system automatically looks for the `SetPointLimit.xlsx` file in the exact same folder where the `AQR_Dashboard_v1.1.0.exe` program is located.

2.2 Data Processing & Calculation Logic (Specific to 5-min Sampling)
•	Header Parsing: A function called `find_header()` searches line by line until it finds the word `<>Date` to skip messy introduction text in the CSV.
•	Deduplication: The system uses a command (`drop_duplicates`) to find and delete any rows that have the exact same Date and Time, keeping only the first one it finds.
•	25-Minute Rule (Strict Continuity):
    •	Sampling Rate: The system expects data every 300 seconds (5 minutes).
    •	Contiguity Check: It enforces continuity using a formula (`diff(5) == 1500`). In plain English, this means the system calculates the time difference between the current reading and the reading from exactly 5 rows ago. If the difference is exactly 1500 seconds (25 minutes), it proves that no data is missing and the excursion is continuous.
    •	Violation Logic: A data point is only flagged as a true "Violation" if it is outside the allowed limits AND it passes the strict 25-minute continuous time check.
•	Pressure Synchronization: To compare room pressure against corridor pressure accurately, the system uses a data-joining technique (`pandas.merge` with an 'inner' join). This forces the system to only compare records that were recorded at the exact same second.
•	Statistics: The system calculates the lowest (`min()`) and highest (`max()`) values over the duration of the continuous violation block.

2.3 Data Transformation Module
•	Input Handling: The system accepts massive bulk files from specific building areas (like Module 5 or the Main Plant).
•	Splitting Logic:
    •	Regex Mapping: It uses pattern-matching codes (Regular Expressions, or "Regex") to automatically hunt for room names in the messy column titles. For example, it looks for patterns like `[12]P\d{3}` to find Module 5 rooms.
    •	Point Mapping: A specific function (`find_point_mapping()`) automatically links 'TEMP', 'HUM', and 'PRES' labels to the correct room.
•	Output: It generates individual, room-specific CSV files that are neatly formatted with standard headers and footers, ready for analysis.

2.4 Security and Data Integrity
•	Secure Audit Trail: The system logs events using a format called JSON Lines. Each line contains a cryptographic signature (SHA-256 hash) that includes the signature of the previous line. When the program starts, a function called `verify_audit_trail()` mathematically checks these signatures to detect if anyone tampered with the file.
•	Checksums: The system calculates unique digital fingerprints (SHA-256) for every CSV file and the `SetPointLimit.xlsx` file.
•	Read-Only: The system is hardcoded to use the `open(file, 'r')` command, where 'r' strictly means Read-Only, making it physically impossible for the software to alter the original data.
