SECTION 2: MODULE DECOMPOSITION
This document describes the software design and specific algorithms used to satisfy the functional specifications. Plain English summaries are included to ensure clarity for non-technical reviewers.

2.2 Calculation Engine (`analysis_logic.py`)
•	Time-Based Windowing Algorithm:
    •	Strict Continuity Check: The engine uses the formula `diff(5).apply(lambda x: x.total_seconds()) == 1500`. This algorithm checks if 6 consecutive data points (which are 5 minutes apart) span exactly 1500 seconds (25 minutes). This is how the system proves an excursion is continuous.
    •	Violation Logic: A data point is only considered part of a "violation block" if the value is out-of-spec AND the 25-minute continuity algorithm confirms the duration.
•	Pressure Synchronization Logic: To compare pressure between two different areas, the engine uses `pandas.merge(..., how='inner', on='DateTime')`. This algorithm strictly matches timestamps, ensuring that the system never compares pressure values that were recorded at different times.

2.3 Data Transformation Module (`app.py` & `analysis_logic.py`)
•	Dynamic Mapping Engine:
    •	`find_point_mapping()`: This function uses text-matching algorithms (Regular Expressions) to automatically scan bulk data files, identify room codes, and match them with their corresponding Temperature, Humidity, or Pressure data columns.
    •	`api_split_reports()`: This is the web traffic controller (Flask route) that receives your command from the user interface, reads the huge bulk file, and coordinates the creation of smaller, individual room files.
•	File Generation:
    •	`_write_custom_csv()`: This function builds the final CSV file, carefully recreating the standard report format (including the required headers and the "End of Report" footer) so the analysis engine can read it correctly.
